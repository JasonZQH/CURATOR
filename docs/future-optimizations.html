Roadmap · Runtime kernel

Future optimizations

The north star is not "multi-agent chat." It is an evidence-driven runtime kernel for agent work — observe, evaluate, route, dispatch, commit, expose — with decision authority kept outside the providers. This document tracks that backlog without turning it into immediate scope.

W What

From a proven Phase 0 to a runtime layer

Curator P0 now proves local init, durable loop ledger storage, typed provider outputs, provider run attribution, context packages, a harness boundary, deterministic runtime decisions, QA-fail retry, pause/resume records, action approval gates, provider profile readiness, unique workflow run ids, a runtime workspace, and snapshot-based Textual panels. Future work should evolve this into a runtime layer that can compile, schedule, dispatch, observe, evaluate, route, revise graphs, and commit durable state around real providers — the loop in Figure 1.

Y Why

Static happy paths are not enough

Loop engineering is useful, but static happy-path workflows are not enough for real software work. Real runs fail, discover missing scope, need reviewer routing, encounter policy limits, require human approval, and must recover after interruption. Without a durable control layer, those decisions become implicit in chat history or provider behavior.

Curator should keep decision authority outside providers. Providers can execute bounded role work, but Curator should own evidence requirements, runtime decisions, retry policy, graph revision, memory updates, and user-visible audit trails.

H How

Treat each category as a research track

Before implementing an item, convert it into a focused design document and implementation plan. Avoid bundling unrelated runtime work into one phase.

Scope guard: the current v0-alpha release remains a local, single-writer workflow. The blueprint is the V1 target — Project/Session/Goal/Task DAGs, Agent and AgentRuntime separation, Provider/Capability registry, append-only events, artifacts and memory manifests, leases/heartbeats, native provider TUI integration, runtime handoff, and bounded parallel workers. These must be documented as migration tracks until the repository implements them; the v0-alpha docs must not imply that they already exist.

Learning PatchesGenerate reviewable proposals to update memory, skills, hooks, or templates after repeated failures.What evidence is required before proposing a patch? Which patches require approval?Turns experience into improvement, but automatic mutation is unsafe.

H Agent memory

Evidence-backed, not chatbot memory

Agent memory should be evidence-backed project state that improves future loops while remaining inspectable, editable, and reversible — never unreviewed chatbot memory.

F Future · considerations · trade-offs

Keep the path, not the framework

Curator should not blindly copy an existing framework. The product opportunity is the runtime decision layer and the evidence ledger around it. The near-term implementation should stay small, but the architecture should preserve the path from single-writer orchestration to richer real-provider scheduling.
