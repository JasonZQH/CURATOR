Curator Future Optimizations

This document tracks future optimization directions for Curator after the real provider runtime foundation. It preserves architecture context, trade-offs, and research questions without turning them into immediate implementation scope.

What

Curator P0 now proves local init, durable loop ledger storage, typed provider outputs, provider run attribution, context packages, a harness boundary, deterministic runtime decisions, QA fail retry, pause/resume records, action approval gates, provider profile readiness, unique workflow run ids, a runtime workspace, and snapshot-based Textual panels. Future work should evolve this into a runtime layer that can compile, schedule, dispatch, observe, evaluate, route, revise graphs, and commit durable state around real providers.

The north star is not only "multi-agent chat." It is an evidence-driven runtime kernel for agent work: 

Observe
  -> collect evidence
  -> evaluate against contracts, policy, budget, and ownership
  -> choose route
  -> dispatch the next role/provider
  -> commit durable state
  -> expose an inspectable snapshot

Why

Loop engineering is useful, but static happy-path workflows are not enough for real software work. Real runs fail, discover missing scope, need reviewer routing, encounter policy limits, require human approval, and must recover after interruption. Without a durable control layer, those decisions become implicit in chat history or provider behavior.

Curator should keep decision authority outside providers. Providers can execute bounded role work, but Curator should own evidence requirements, runtime decisions, retry policy, graph revision, memory updates, and user-visible audit trails.

How

Treat each category as a backlog and research track. Before implementing an item, convert it into a focused design document and implementation plan. Avoid bundling unrelated runtime work into one phase.

Scope guard: real provider execution, runtime dynamic role dispatch, executable graph revision, ownership, parallel work, memory learning, and skill runtime are P1/MVP tracks. P0 already has durable pause records, resume events, scope-change draft creation, provider profile records, and action approval gates; those foundations should be extended rather than redesigned.

Optimization Categories

Category Future Direction Research Questions Trade-offs
Runtime Evaluator Evaluate evidence against done criteria, contracts, risk, budget, and approval rules. Which checks are objective gates, and which require an agent critic? How should confidence be represented? Improves correctness, but adds complexity and must remain explainable.
Runtime Role Dispatch Select the next role from evidence and failure state, not only compile-time task signals. When should QA, security, release, PM, or human review be selected dynamically? Enables adaptive workflows, but can create surprising routes without clear audit records.
Graph Revision Promote paused scope-change drafts into accepted revisions and executable scheduler paths. Which completed tasks are inherited? What requires user approval? How is impact previewed before acceptance? Essential for autonomy, but risky without strong review UX.
Ownership Model Track ownership of tasks, evidence, files, worktrees, failures, and decisions. Who owns a failed diff? Which role retries? How are reviewer findings assigned? Needed before parallel work, but too heavy for the current single-writer runtime.
Durable Recovery Turn durable resume events into safe continuation, rerun, or cancellation behavior. What should resume from SQLite? What should rerun? How are provider-native sessions and context packages reconciled? Raises reliability, but provider-specific behavior will shape the design.
Parallel Scheduling Run independent tasks concurrently after ownership and conflict rules exist. How are worktrees isolated? How are patches merged? Which QA validates combined output? Improves throughput, but multiplies merge, routing, and recovery complexity.
Provider Adapters Connect Codex and Claude Code through bounded harness runs, context packages, action policies, provider profiles, and capability manifests. Which outputs are machine-readable? How are permissions, tools, MCP servers, and artifacts mapped? Required for MVP value, but must not compromise Curator-owned state boundaries.
Agent Memory Evolve project, role, session, and learning memory into evidence-backed state. What becomes long-term fact? What remains session summary? How are stale facts corrected? Improves future loops, but bad memory can permanently degrade behavior.
Learning Patches Generate reviewable proposals to update memory, skills, hooks, or templates after repeated failures. What evidence is required before proposing a patch? Which patches require approval? Turns execution experience into improvement, but automatic mutation is unsafe.
Skills / Capability Packs Load reusable operating manuals such as CI triage, security review, OAuth validation, or release checklist. Which skills are packaged, project-defined, or learning-patch-proposed? Strengthens agents without retraining, but stale guidance and injection risks remain.
Hooks Add controlled hook points such as before_run, after_run, on_failure, on_evidence, and on_stop. Which hooks are safe? How are side effects audited? Can hooks be disabled per project? Improves extensibility, but can become hidden control flow if not governed.
Security / Policy Kernel Extend the current action policy from file/shell/remote VCS gates into network, credentials, dependency install, and merge permissions. How are remembered permissions scoped? How is risk displayed to the user? Which approvals can expire? Security is core to trust, but too much policy early can slow development.
Observability Extend the current workbench and snapshot panels with timeline, decisions, provider streams, usage, failures, and memory patches. What is visible by default? What is expandable? How does a user inspect a failed loop quickly? Improves trust and debugging, but too much UI detail can overwhelm users.

Agent Memory Optimization

Agent memory should not become unreviewed chatbot memory. It should be evidence-backed project state that improves future loops while remaining inspectable, editable, and reversible. 

Memory layers
  Project memory
    - what the project is
    - architecture facts
    - stable commands

  Role memory
    - role-specific lessons
    - user corrections for that role
    - common mistakes

  Session memory
    - what happened in one loop
    - evidence and decisions
    - final summary

  Learning patches
    - proposed updates from repeated failures
    - require approval before long-term write

  Skill memory
    - reusable operating manuals
    - versioned through reviewable patches
Design rule: execution can propose memory changes, but it should not silently rewrite long-term memory. Evidence links and approval gates are required for durable memory updates.

Future improvements/considerations/trade-offs

Curator should not blindly copy an existing framework. The product opportunity is the runtime decision layer and the evidence ledger around it. The near-term implementation should stay small, but the architecture should preserve the path from single-writer orchestration to richer real provider scheduling.
