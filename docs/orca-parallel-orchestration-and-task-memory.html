Design · Parallelism & memory

并行编排与任务记忆

Orca 的并行能力本质是"worktree 隔离 + terminal dispatch + 协调消息"，其记忆本质是 "尊重 provider 仓库记忆 + 持久化运行协作状态"。Curator 应借鉴前者的可操作性，但把后者 升级为有来源、有作用域、可过期、可验证的任务记忆 —— 不把聊天记录或 agent 自述当作长期事实。

状态：提议，作为 Curator V1 并行编排与任务记忆的设计输入。
两条设计判断
  1. 并行的基本隔离单位应是 Workspace / Worktree，而不是"共享目录中的多个 agent 进程"。
  2. 任务记忆的基本写入单位应是附带证据引用的 TaskMemoryEntry，而不是 transcript、自由文本 checkpoint 或模型总结。
W What

范围与事实来源

本文把 Orca 已公开的并行编排、worktree、checkpoint、hooks 和 memory 机制拆解为可复用 artifact， 并定义 Curator 应实现的最小模型、状态流、记忆规则、阶段计划和验收标准。它不把 Orca 的桌面 IDE 功能 当作前置条件，也不宣称 Curator 当前已具备并行、DAG 或远程运行时能力。

  • Orca 的公开文档说明其任务有 Task、Dispatch、持久消息、心跳和 decision gate；该编排层仍被标为 experimental。
  • Orca 以独立 git worktree 承载并行 agent；同一提示可以在多个 worktree 中竞赛，随后人工比较 diff 并选择结果。
  • Orca 对 memory 采取 provider-owned 策略：读取且不改写 AGENTS.md / CLAUDE.md；另外提供可持久化的 worktree checkpoint 与 hook endpoint。
  • Curator 当前已具备 SQLite ledger、单写入锁、验证证据、fresh-context reviewer、暂停/恢复和失败记忆；当前范围明确为顺序单写入者。
H How · Orca 的并行编排模型

worktree 隔离 + dispatch + 协调

ArtifactOrca 的职责并行价值Curator 采纳方式
Worktree一个任务对应一个物理 git worktree，并记录 base/start-from ref。让多个 writer 不共享文件系统写入面；同题竞赛与独立子任务都可安全执行。新增 Workspace，绑定 repo、base ref、branch、path、owner task 与生命周期；它是并行的唯一写入边界。
Task拥有 spec、依赖和 pending → ready → dispatched → completed/failed/blocked 状态。使 coordinator 不必依赖 terminal transcript 判断谁能开始。把现有编译步骤迁移为持久 Task 与 TaskDependency；状态转换只允许 scheduler 执行。
Dispatch把 Task 分配给一个 terminal；重试获得新的 dispatch context。避免旧 worker 的迟到消息完成新的重试任务。新增不可变 Execution；所有 heartbeat、完成声明、产物和失败都必须携带 task_id + execution_id。
消息与 heartbeat持久化 status、dispatch、worker_done、escalation、decision_gate、heartbeat 等通信。让 coordinator 能等待正确事件，而不是轮询所有终端。新增 append-only CoordinationEvent，但消息只改变"观察状态"；真正完成仍需验证 evidence。
Decision gate阻塞某个 Task，直到 coordinator 或人类记录决策。把依赖于选择的问题从临时聊天移入可恢复状态。复用并扩展当前 approval/pause 模型：gate 需要问题、选项、影响任务、决策者、有效期和可审计 resolution。
Diff review比较各 worktree 的 diff，行级批注可批量发回 agent。把"选择胜者"从主观 terminal 阅读变为可操作评审。将 finding 作为带文件/行范围的 ReviewFinding evidence，并在定向 resume 时注入 writer context。

Orca 的有效交互模式

  1. Fan-out：同一任务在多个 worktree 竞赛，随后比较 diff。
  2. Decompose：coordinator 创建有依赖的 task，再向可用 terminal dispatch ready task。
  3. Escalate：worker 通过 coordinator 问题与 gate 获取阻塞决策，而不是卡在本地交互提示。
  4. Observe：工作中、等待输入、空闲等 agent 状态被呈现为全局可见的运行态，而非仅可从某个 terminal 猜测。
不能照搬的语义：Orca 的 worker_done 是协调协议中的完成通知，不等于可交付完成。Curator 必须保持"worker_done → 采集 artifact → deterministic validation → reviewer / human gate → scheduler transition"的顺序。
H How · Orca 的任务记忆分层

三层并存的记忆机制

Orca 没有公开一个统一的、证据化的 semantic task-memory 数据模型。它实际采用三层并存的记忆/上下文机制：

层载体所有者适合存什么局限
Provider 指令记忆AGENTS.md、CLAUDE.md 与 provider 配置用户 / agent provider仓库约束、编码约定、执行指令。不是 task-specific 事实；缺少执行 provenance；不应由编排器静默改写。
运行协作记忆Task、Dispatch、消息、heartbeat、gate、session restoreOrca runtime谁在做什么、当前阻塞、派发版本、待决问题。主要是操作状态，并不自动证明结论或代码质量。
工作区摘要记忆worktree checkpoint / comment人和 agent 协作当前进展、最近发现、下一步。自由文本可过期、可覆盖；必须先读后写，仍不宜作为长期事实来源。
自动化连续上下文可复用的 automation sessionOrca runtime周期性工作中的短期会话连续性。会话连续不等于可靠的项目记忆；应可切换为 fresh session。

对 Curator 的关键启发：尊重 provider-owned memory，但不要把它误认为 Curator memory。Curator 应读取并打包 AGENTS.md / CLAUDE.md 等外部指令，记录其内容 hash 和读取范围；这些文件保持用户/agent 所有。Curator 自己维护的任务记忆必须独立、可查证、可撤销。

H How · Curator 的目标 Artifact 模型

把并行升级为可回放链

当前本体成熟度：Phase 0 → V1

Curator 当前已是 ontology-aware 的 Phase 0：拥有 Goal、Session、Loop、Task、Role、Provider Run、Evidence、Decision、Pause 和 Memory 等实体。当前运行时仍是顺序单写入者，而非完整的并行 Agent Runtime Ontology。

本体边界当前 Phase 0V1 必需补足
Agent 身份Role slot、provider profile 与一次 provider run 已有，但语义仍部分重叠。明确 Agent、AgentRuntime 与 Execution。
任务关系编译后的顺序步骤可执行，但没有持久 dependency graph。TaskDependency、ready queue、task ownership、冲突检测与 bounded concurrency。
执行身份Provider run 保存了调用结果，但没有完整的 runtime/process/workspace 归属。每次 dispatch 或 retry 新建不可变 Execution；旧 execution 只读保留。
写入边界通过 clean baseline 与 project lock 保护单一项目目录。Workspace/Worktree 实体、独占 lease、base ref、branch、资源冲突规则。
证据与记忆已有 evidence ref，以及针对 retry/failure/pause 的 memory distillation。一等 ArtifactRef、ReviewFinding、TaskMemoryEntry，并加入 provenance、scope、confidence、TTL、supersedes。
权限与能力已有 action policy 和 approval request。CapabilityGrant 显式绑定 provider/runtime/execution 与各类访问范围。
三层边界不可混淆：Agent 是可选择的工作主体及其能力；AgentRuntime 是存活的 provider 进程/终端实体；Execution 是某个 Task 的一次不可变尝试。同一 provider 可支撑多个 runtime，同一 runtime 可发生多个 execution；失败重试必须产生新的 execution，而不能覆盖旧记录。

新增 Artifact 的字段与规则

新 Artifact最小字段写入规则读取规则
Workspaceid、repo、base_ref、branch、path、owner_task_id、status、leasescheduler 创建；一个可写 Task 只持有一个独占 workspace。worker 只获得其 workspace；reviewer 默认只读。
Executionid、task_id、attempt、provider/profile、workspace_id、started/stopped、reason每次 dispatch/retry 新建，不覆盖旧 execution。所有事件、产物和 memory 必须可回溯到 execution。
TaskCheckpointtask_id、execution_id、phase、summary、blocker、next_step、supersedesagent 或人可提议；系统保存不可变版本并标记当前版本。仅用于运行观察和短期 context，不能单独导致任务完成。
TaskMemoryEntryscope、kind、claim、evidence_refs、confidence、ttl、status、created_by由 verifier/reviewer/scheduler 生成，或由 agent 提议后经验证/审批写入。按 project/task/component/provider scope 和有效期检索；必须显示来源。

DecisionGatequestion、options、affected_tasks、owner、resolution、expiryscheduler/coordinator 创建，授权人解决。只释放受影响的 ready task；resolution 进入 evidence-linked memory。

记忆不变量

  1. 不把 transcript 当 memory。原始输出是 artifact；摘要只是引用 artifact 的派生对象。
  2. 不让 worker 自证完成。worker_done、checkpoint 与 hypothesis 只能触发后续验证。
  3. 每条长期记忆可追溯。必须能打开 source execution、artifact hash、decision 或 human approver。
  4. 记忆有作用域与淘汰策略。Task 级总结不得自动污染 Project 级知识；低置信度与失败假设会过期。
  5. 外部记忆文件不被静默修改。AGENTS.md / CLAUDE.md 是 provider-owned；Curator 只记录读取版本，修改必须走显式 proposal 与 approval。
Y Why

worktree 之外还需语义与证据

仅使用 worktree 可以避免文件冲突，却不能避免语义冲突：两个 task 可能修改同一个 API，或在错误假设上平行推进。仅使用 agent chat 可以快速协作，却不能恢复、验证或解释决策。把 worktree ownership、Task/Execution 身份、证据与记忆合并成一条可回放链，才能让并行提高吞吐而不降低交付可信度。

借鉴如何改造成 Curator 优势
独立 worktree 并行为每个写任务建立可审计 ownership、base ref 和 lease；合并前必须执行组合验证。
Task / Dispatch / heartbeat保留交互协议，但所有最终状态由 scheduler 与 evidence 决定，而非消息决定。
checkpoint将自由文本提升为可版本化的 TaskCheckpoint，显式标记 superseded 与关联 execution。
provider memory files只读纳入 context package，并记录 hash；Curator 的长期 memory 与 provider 指令分开管理。
agent race + diff selection把候选 diff、验证结果、review findings 统一成可比较 evidence pack，而不是凭终端观感选胜者。
F Future · considerations · trade-offs

建议实施顺序

  1. V1.1 Canonical artifacts：新增 Task、Dependency、Workspace、Execution、ArtifactRef 和 append-only CoordinationEvent；先提供 Phase 0 ledger 的只读映射。
  2. V1.2 Worktree adapter：实现 create/inspect/archive worktree、clean baseline、base-ref pinning、独占 lease 和 crash reconciliation；此时仍只允许一个 writer。
  3. V1.3 Ready queue：仅调度无依赖且 workspace 不冲突的 task；上线 bounded concurrency，初始上限为 2，稳定后才提高到 4。
  4. V1.4 Evidence-backed task memory：实现 TaskCheckpoint、TaskMemoryEntry、ReviewFinding、provenance/TTL/retrieval；把现有 retry/failure/pause memory 迁移为该模型。
  5. V1.5 Control Desk：展示 DAG、workspace、execution、heartbeat、blocker、evidence、memory 与 gate；再考虑 provider-native TUI attach、通知和远程 runtime。

MVP 验收标准

  • 两个独立 task 可在独立 worktree 中并行运行；同一路径或声明共享资源的 task 不会同时获得写 lease。
  • worker crash、app restart、lease 过期后，旧 Execution 不会继续写入；恢复会创建新的 Execution 并保留历史。
  • 迟到的 worker_done 无法完成新的 retry attempt。
  • Task 只有在要求的 validation evidence、review 状态与必要 gate 都满足后才能进入 DONE。
  • 每条 validated_fact 与 decision memory 都能定位到 source artifact/decision；过期或 superseded memory 不进入默认 context package。
  • 任意 provider-owned 指令文件的写入均要求显式 approval，并在 ledger 中有 before/after hash。

明确延后

  • 完整 Electron IDE、无限终端分屏、Monaco 编辑器和移动端不是并行 scheduler 的前置条件。
  • SSH/远程 runtime 应在 lease、process identity、recovery、密钥与网络权限模型稳定后再实现。
  • 不要以"支持任意 CLI agent"为近期目标；优先完成 capability probe、profile contract 和安全策略矩阵。
  • 不要自动把 agent reflection 写入长期 memory；应先生成 proposal，再由 evidence 或人类审批提升其作用域。
