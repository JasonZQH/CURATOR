v0.1.0 发布计划 v3

结论：可以开始实现，但 v0.1.0 必须继续作为单机、单写入者、顺序工作流的 Phase 0 发布。画布中的 DAG、并行 worker、AgentRuntime、Provider TUI、Capability Registry 和 Runtime Handoff 是 V1 迁移目标，不能提前写成已实现能力。

W What

两个交付对象

v0.1.0 的交付对象是一个可恢复、可审计的本地控制面：Goal → writer provider → deterministic verifier → fresh-context reviewer → human confirmation。状态以 SQLite ledger 为准，provider 只执行受边界约束的工作，shell/TUI 只提供入口和观察面。

V1 的交付对象才是画布中的多角色运行时：Project/Session/Goal/Task DAG、Agent 与 AgentRuntime 分离、Execution 与 ExecutionRuntime 分离、Provider/Capability 管理、 Artifact/Memory 文件清单、lease/heartbeat、Control Desk 与原生 Provider TUI 协作，以及 最多四个本地并行 worker。

H How

一、v0.1.0 必须完成的发布门槛

优先级工作项验收条件
P0修正事务状态实现不在标准 sqlite3.Connection 实例上挂载自定义属性。改用 connection subclass/factory、外部 transaction state 或显式事务 API；测试 nested transaction、rollback、异常恢复和并发写入。
P0修正重复崩溃后的 pause/recovery同一个 loop 在恢复后再次崩溃，必须生成新的 recovery attempt/open pause，不能因复用已 RESOLVED 的固定 pause id 而把 loop 留在 PAUSED 且没有可恢复入口。
P0统一所有写入口的单写入者锁shell、TUI、CLI 直接调用 start_goal_loop 或 resume_goal_loop 都经过同一把 project/runtime lock；锁不可用时只允许只读查看，禁止 approve/resume/retry/cancel 等 mutation。
P0完成进程组和中断语义provider 启动使用 process group；Ctrl+C、timeout、provider error、machine restart 都能记录 stopped_by/stop_reason，清理子进程，不留下 RUNNING execution。
P1收紧 provider 输入输出边界prompt/context 优先通过文件或 stdin 传递，避免把敏感内容放进 argv；事件、日志、response、evidence、artifact ref 的字段与 ledger 类型一致。若 OUTPUT_CHUNK 不落 ledger，就不能宣称"所有 provider event 已持久化"。
P1发布与安装验收CI 同时 smoke-test wheel 和 sdist；tag 触发条件、macOS 测试、Python 版本、干净安装、provider 缺失时的阻塞提示都能复现。文档中的安装 tag 必须与实际发布 tag 一致。
P1文档事实对齐README 与运行时文档明确标注 Phase 0/V1；命令示例使用稳定 slot 名称；不把 DAG、并行、tmux、heartbeat、native TUI 或 handoff 写成当前能力。

二、.sh 可以承担的工作

.sh 适合作为 OS/process adapter：检测命令和版本、创建或 attach tmux、启动 provider、收集 stdout/stderr、转发 signal、生成日志或 artifact manifest。Python 仍然负责 事务、状态机、锁、审批、retry、recovery、事件索引和最终决策。shell adapter 不得直接改 ledger、绕过权限、决定 task ownership，也不得把 agent A 的输出直接 pipe 给 agent B 而跳过 事件链。

最小契约应固定 context-file/workspace 输入、NDJSON 或 manifest 输出、stdout/stderr 分离、 process-group 信号转发和 exit code。这样 v0.1.0 可以继续使用 direct subprocess，V1 再将同一 adapter 接入 tmux/Ghostty/Warp，而不把运行时状态散落进 shell 逻辑。

Y Why

分层发布的理由

画布的核心不是"把更多 agent 接进来"，而是让业务执行身份、provider 运行身份、审批、证据和恢复都 可持久化。若把 v0-alpha 直接改写成 V1，用户会误以为已经具备依赖调度和并发隔离；若只修几个 stale 字符串，又会遗漏 Execution/Runtime/Handoff 的语义变化。分层发布能同时保持 v0.1.0 可交付和 V1 架构可演进。

F Future · considerations · trade-offs

迁移取舍

  • 优先建立 v0 ledger 到 V1 canonical model 的只读映射，再做破坏性迁移。
  • 在并行前先实现 workspace lock/ownership；否则"并行"只是并发制造脏 diff。
  • Capability policy 要区分 filesystem、shell、network、credential 和 external access，不能只用一个 allow/deny。
  • 大日志、patch、prompt snapshot 和测试结果放 filesystem，SQLite 只保存索引、hash、状态和引用。
  • 每个新函数继续遵守项目约定：用一行命令式注释说明函数职责；新增文档保持 What/How/Why/Future 四段结构。
