Shell Adapter Boundary

What

Shell scripts are a useful operating-system adapter for launching a provider, checking local prerequisites, creating or attaching a tmux session, and collecting filesystem artifacts. They are not the authoritative orchestration layer. SQLite state, task transitions, approval decisions, leases, recovery, and audit events remain owned by the Python control plane.

How

A shell adapter should be a thin, testable boundary with an explicit contract: receive a context-file path and workspace path, start one bounded provider/runtime process, stream structured events or write an event manifest, preserve stdout and stderr separately, forward signals to the process group, and return documented exit codes. Python commits the corresponding event and state changes in a transaction. 

Python control plane
  -> validates policy, workspace, capability, and lease
  -> writes immutable input snapshot
  -> invokes adapter.sh with context and runtime references
adapter.sh
  -> starts provider or tmux-backed runtime
  -> emits NDJSON/events and exits with a documented code
Python control plane
  -> records provider/runtime events, artifacts, failure, retry, or handoff
  -> decides the next task state

The adapter must not silently mutate the ledger, decide task ownership, bypass approval policy, or chain directly into another agent. Direct shell-to-shell chaining would make failures and permissions invisible to the scheduler. Project-provided scripts also need an explicit trust decision; they must not be executed merely because they are present in the workspace.

Why

This split uses the strength of each layer. Shell is good at portable process and terminal operations, while Python is better suited to typed contracts, SQLite transactions, concurrent scheduling, structured recovery, and provider-independent policy. It also lets the same provider adapter work from a direct subprocess in Phase 0 and from tmux/Ghostty/Warp runtimes in V1.

Future improvements/considerations/trade-offs

  • Define stable exit codes for completed, provider error, approval required, retryable failure, and user interruption.
  • Use context files or stdin instead of putting prompts and secrets in process arguments.
  • Provide macOS/Linux compatibility checks for flock, process groups, tmux, and signal behavior.
  • Add contract tests with fake providers before enabling native TUI focus and handoff.
  • Keep shell scripts small; move stateful policy and recovery logic into Python services.
