Provider, Agent, and Runtime Identity

What

Phase 0 separates role slots such as writer.default from provider profiles such as codex-work or claude-team. The blueprint makes this boundary explicit with four identities: an Agent is a configured role worker, a Provider is an installed/backend identity, an AgentRuntime is one live process or provider TUI session, and an Execution is the durable task attempt that survives runtime handoff.

How

Provider profiles are stored in SQLite with a non-secret credential_ref. Phase 0 role slots bind to profiles through role_provider_bindings. The target migration adds provider status, capability/version detection, installation records, scoped permissions, runtime handles, heartbeats, and execution-runtime history without exposing credentials in prompts, logs, or shared memory. 

/providers
/agent bind writer.default codex-work
/agent switch writer.default claude-team

The runtime workbench renders provider profile status, latest quota state, and active slot bindings. The scheduler requires real provider drivers for user work, and provider objects expose provider_name, provider_profile_id, and provider_session_id for ledger attribution. These are Phase 0 compatibility fields, not the complete target runtime registry.

Current CLI commands render and switch existing provider profile records. Dedicated profile creation, credential setup, automatic quota refresh, and provider-specific session bootstrapping are intentionally not user-facing flows yet.

Why

Account identity, capability availability, live process identity, and task business identity are different concerns. Keeping them separate lets the scheduler switch or recover a runtime without changing the task's execution identity, while preserving a complete audit trail of provider and capability use.

Future improvements/considerations/trade-offs

This version intentionally avoids storing raw credentials and does not call external quota APIs. Future work can add Provider Adapter contracts, capability installation/status, Keychain-backed credentials, native TUI/tmux runtimes, heartbeat-based health, and scheduler-level reassignment when a provider or capability is unavailable.
