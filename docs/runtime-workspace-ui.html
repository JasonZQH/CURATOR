Runtime Workspace UI

What

The Runtime Workspace UI makes the runtime graph the center of the user experience. Conversation remains an input, but the primary objects are runtime state, agents, provider profiles, queue items, evidence, events, pauses, and next actions.

How

The CLI uses /workbench as the main runtime view. Supporting commands expose focused slices of state without requiring the user to inspect SQLite or remember hidden configuration.

The shell workspace reads the SQLite runtime ledger directly. The Textual TUI renders separate WorkflowSnapshot panels for Runtime, Active Roles, Providers, Evidence, and Events. This keeps provider/tool output out of the conversation stream and makes runtime state easier to scan.

Why

AgenticOS is a runtime, not only a chat interface. Users need to know which role is active, which provider profile is bound, whether quota is healthy, what evidence exists, and which command will move the system forward. This mirrors the useful parts of Hermes-style agent observability while keeping the AgenticOS center of gravity on runtime state.

Future improvements/considerations/trade-offs

This version is snapshot-based. Future work can add streaming runtime events, interrupt-and-redirect controls, command palette overlays, session switching, provider setup flows, and richer DAG visualization. Provider switching remains manual by design; automatic switching should require an explicit policy, quota source, approval model, and trust boundary.
