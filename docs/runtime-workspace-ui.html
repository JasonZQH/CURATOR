Runtime · Surfaces

Control Desk & provider TUI

The supervisory surface and the execution surface are two different things. Curator's workbench reads the ledger and tells you what is happening; the provider's own TUI is where native approvals and interactive input actually happen. The ledger stays the system of record either way.

W What

A workbench today, a Control Desk next

The current UI is a Phase 0 runtime workbench over a SQLite ledger. The target Control Desk in the blueprint is a supervisory surface for Project, Session, Goal, Task, Execution, Agent, Provider, Artifact, Approval, and Event state. It does not replace the provider's native interactive TUI.

H How

Commands and panels over the ledger

The CLI uses /workbench as the main Phase 0 view. Stable slot IDs such as writer.default and reviewer.default are the compatibility vocabulary for the current implementation. Supporting commands expose focused slices of state without requiring the user to inspect SQLite. 

/workbench                         /queue
/agents                            /queue tick
/providers                         /approvals
/agent status writer.default       /approve approval-id optional message
/agent bind writer.default codex   /reject approval-id optional message
/agent switch writer.default team  /evidence · /node current
/resume continue with this narrower fix

The shell workspace reads the SQLite runtime ledger directly. The Textual TUI renders separate WorkflowSnapshot panels for Runtime, Active Roles, Providers, Evidence, and Events, and shows live escaped provider output while a run is active.

Transcript is not the record: live transcript text is temporary UI output and is not replayed. Lifecycle, tool, and bounded OUTPUT_CHUNK events are ledgered. In the target model these panels become the Control Desk and add task dependencies, worker leases, heartbeat age, approval requests, artifact references, and blockers. The Claude Code or Codex TUI remains where native approval and interactive input are handled; the Control Desk only focuses that surface and synchronizes the resulting state.
Y Why

Supervisory state must outlive the transcript

Users need to know which task and runtime are active, which provider capability is being invoked, whether an approval or lease is blocking progress, what evidence exists, and which action will move the system forward. Keeping this supervisory state outside provider transcripts is required for auditability and recovery.

F Future · considerations · trade-offs

What waits for the trust boundary

Phase 0 remains sequential and single-writer. Future work can add task DAG visualization, lease/heartbeat views, tmux-backed provider windows, runtime handoff snapshots, and capability approval workflows. Automatic provider switching should wait for capability detection, quota sources, explicit approval policy, and a tested trust boundary.
