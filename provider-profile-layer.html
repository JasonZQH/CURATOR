Provider Profile Layer

What

The Provider Profile Layer separates role instances such as engineer.1 from provider accounts such as codex-work or claude-team. A role instance still represents work capability, while a provider profile represents the execution identity, credential reference, session, and quota state.

How

Provider profiles are stored in SQLite with a non-secret credential_ref. Role instances bind to profiles through role_provider_bindings. Provider executions can record provider_profile_id and provider_session_id in the provider run ledger. 

/providers
/agent bind engineer.1 codex-work
/agent switch engineer.1 claude-team

The runtime workbench renders provider profile status, latest quota state, and active agent bindings. The scheduler now requires real provider drivers for user work, and provider objects expose provider_name, provider_profile_id, and provider_session_id for ledger attribution.

Current CLI commands render and switch existing provider profile records. Dedicated profile creation, credential setup, automatic quota refresh, and provider-specific session bootstrapping are intentionally not user-facing flows yet.

Why

Account identity and quota are resource concerns, not role concerns. Keeping them separate lets the scheduler pause or switch provider identities without changing the role model, work queue, or context package reconstruction path.

Future Improvements, Considerations, and Trade-offs

This version intentionally avoids storing raw credentials and does not call external quota APIs. Future work can add real Codex and Claude adapters, profile setup commands, automatic quota observation, safer credential backends such as Keychain, and scheduler-level work reassignment when a provider profile becomes quota exhausted.
